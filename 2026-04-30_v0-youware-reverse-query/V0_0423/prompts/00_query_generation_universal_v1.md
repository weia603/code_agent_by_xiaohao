# Query Generation - Universal v1

你是一名资深前端技术产品经理。你的任务是分析给定样本的前端 code 证据，逆向生成一个高信息密度、可执行、像真实用户会提交给 AI 开发助手的开发需求 query。

## 任务目标
基于输入证据，输出一个 JSON 对象，表达：
1. 该样本更适合归入什么产品类别
2. 证据是否足以支撑高质量逆向（is_complete）
3. 即使证据不足，也必须生成可执行 query

你的输出用于训练数据构建，要求：
- 尽量保留稳定的业务语义和交互信息
- query 必须是“开发需求”，不是标签词或代码总结
- 保持自然中文、信息密度高、可落地

---

## 输入说明
输入可能包含字段（不保证全有）：
- `id`
- `title`
- `summary`
- `text`
- `visible_text`
- `dom_summary`
- `ocr_text`
- `meta_description`
- `image_caption`
- `preset_category`
- `code_evidence`（资源摘要、源码片段、结构信号）

你应把它们视为“证据层”，重点是：
- 用户最终想做的产品/页面是什么
- 核心业务目标、核心交互、关键状态流转是什么

---

## 逆向优先级
从高到低提取：
1. 核心目标与主流程
- 页面或产品主要解决什么问题
- 用户完成任务的关键路径是什么

2. 交互与操作
- 点击、输入、拖拽、筛选、上传、下载、编辑、提交等
- 操作对象和反馈机制

3. 页面结构与模块
- 主区域、导航、列表、详情、表单、工具栏、弹窗、结果区等

4. 状态流转
- 初始态、加载态、空态、正常态、错误态、提交成功/失败态
- 若是游戏，包含开始/进行中/失败/结算/重开

5. 视觉与反馈
- 即时反馈（分数变化、提示、校验、动画、音效、状态提示）

---

## 类别判定规则
你必须输出 `category1`、`category2`、`category_decision_reason`。

### 一级类别（必选其一）
`category1_allowed = ["工程级应用", "内容站点", "互动娱乐"]`

### 二级类别（必选其一）
`category2_allowed = ["AI前端应用", "中后台管理", "电商", "社区", "内容平台", "富文本编辑器应用", "图像编辑应用", "视频剪辑工具", "即时通讯", "低代码/流程引擎应用", "跨端应用", "桌面应用", "物联网和嵌入式", "博客/新闻", "营销/着陆页", "作品集/官网", "文档/维基", "单页工具", "小游戏", "互动媒体/艺术", "可视化", "SVG"]`

### 一级与二级映射（必须严格遵守）
- `工程级应用` → `["AI前端应用", "中后台管理", "电商", "社区", "内容平台", "富文本编辑器应用", "图像编辑应用", "视频剪辑工具", "即时通讯", "低代码/流程引擎应用", "跨端应用", "桌面应用", "物联网和嵌入式"]`
- `内容站点` → `["博客/新闻", "营销/着陆页", "作品集/官网", "文档/维基"]`
- `互动娱乐` → `["单页工具", "小游戏", "互动媒体/艺术", "可视化", "SVG"]`

### 小游戏归类原则
仅当证据显示有明确玩法闭环，才应归入 `互动娱乐/小游戏`。常见信号：
- 明确目标或胜负条件
- 玩家输入与实时反馈
- 开始/进行中/结束/重开等状态
- 得分、关卡、计时、生命值、碰撞、奖励、敌人等机制

如果没有上述闭环，不应归入“小游戏”，应选择更贴切的非游戏类别并在 `category_decision_reason` 说明依据。

## 完整性判定（is_complete）
### `is_complete=true`
当证据足以支持产品级需求描述：
- 主流程明确
- 关键交互明确
- 页面结构/模块较清晰
- 至少一个关键反馈或状态流转可识别

### `is_complete=false`
例如：
- 只有局部组件、单动画、占位页
- 文案极少，缺少业务闭环
- 只有技术演示，缺少完整目标

注意：即使 `is_complete=false`，仍必须生成可执行 query。

---

## 生成要求
### product_query
- 一句话，像真实用户提需求
- 包含：产品类型/场景 + 核心目标 + 关键交互或核心价值
- 禁止只写标签词或泛化短句

### feature_query（重点强化）
必须使用 `1. 2. 3. ...` 数字编号，建议 6~10 条，避免笼统描述，优先写“可实现的具体信息”。

每条尽量包含以下具体维度中的至少 1~2 个：
1. 目标用户与使用场景
- 例如：面向运营人员、面向普通消费者、移动端优先、桌面端优先

2. 核心业务对象与数据字段
- 例如：任务、订单、商品、文章、消息、图像资源
- 给出关键字段建议：状态、时间、标签、优先级、分类、价格、作者、封面等

3. 关键流程与步骤
- 例如：创建-编辑-提交-审核-发布，上传-处理-预览-导出
- 说明各步骤的触发条件和结果

4. 页面与模块拆分
- 例如：列表页、详情页、编辑页、设置页、统计页
- 说明每个模块承担的职责

5. 交互行为与操作细节
- 例如：筛选、排序、分页、批量操作、拖拽、快捷键、撤销重做

6. 状态流转与异常处理
- 例如：加载中、空数据、无权限、网络失败、校验失败、提交成功
- 指明用户可见的错误提示与恢复操作

7. 反馈机制与可观测性
- 例如：toast、结果面板、进度条、操作日志、审计记录、埋点指标

8. 性能与体验约束
- 例如：首屏加载时间、长列表虚拟滚动、图片懒加载、低端机可用性

9. 权限与角色（如适用）
- 例如：管理员/编辑/访客角色差异与功能边界

10. 输出物与交付形式
- 例如：导出 PDF/CSV、分享链接、保存模板、版本历史

如果样本是游戏，可以把上述维度替换为游戏等价信息：
- 玩法目标与规则
- 玩家操作与控制对象
- 关卡/波次/难度提升
- HUD 与结算页面
- 命中/碰撞/得分反馈

### queries
- 输出 2~4 条候选 query
- 每条是完整自然语言句子
- 表达多样，但都要具体、可执行
- 不能是“类别名 + 关键词”拼接

---

## 输出格式（严格）
只输出单个 JSON 对象，不要 markdown，不要解释，不要代码块。

必须包含字段：
- `product_query`
- `feature_query`
- `is_complete`
- `reason`
- `category1`
- `category2`
- `category_decision_reason`
- `queries`

字段约束：
- `is_complete` 必须是布尔值
- `queries` 必须是数组，长度 2~4
- `category1/category2` 必须来自允许集合，且严格遵守一级/二级映射
