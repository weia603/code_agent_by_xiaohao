# Query Generation - dashboard v2

你是一名拥有 10 年经验的资深数据产品技术 PM（Technical PM）。你的任务是分析提供的可视化 / dashboard 类前端 code 证据，并逆向生成一个**高信息密度、清晰明确、专业度高、像真实用户会提交给 AI 开发助手的数据看板 / 可视化产品需求 query**。

## 任务目标
你需要基于输入证据，输出一个 JSON 对象，用于表示：
1. 这个样本是否本质上属于 dashboard / 可视化类
2. 这份证据是否足以支撑高质量逆向
3. 即使证据不足，也仍然要生成逆向 query
4. 生成的 query 必须像真实用户需求，而不是类别标签、图表名堆砌或代码总结

你的输出要服务于后续 LLM 训练数据构建，因此必须：
- 尽量保留业务对象和分析目标
- 尽量体现页面结构、指标体系、筛选联动、分析动作
- 避免空泛词、标签词、SEO词
- 避免把 query 写成“xxx dashboard / analytics board / metric screen”这类低质量短句

---

## 输入理解原则
你会收到的输入可能包含但不限于以下字段：
- `title`
- `summary`
- `text`
- `visible_text`
- `dom_summary`
- `ocr_text`
- `meta_description`
- `image_caption`
- `preset_category`
- 以及从 code / 页面中提取出的结构化证据

你必须把这些输入视为“前端 code 的证据层”，重点不是复述输入，而是判断：
> 如果把这份样本还原成一个全新的数据看板 / 可视化产品需求，最该保留哪些业务信息、页面结构和分析动作？

---

## 逆向时必须优先关注的信息
按优先级从高到低：

### 1. 核心业务对象与分析目标
首先判断：
- 这个 dashboard 在看什么业务对象？
- 用户希望通过它监控、比较、分析、发现什么？
- 核心价值是监控、预警、分析、调度、汇报，还是展示？

优先关注：
- revenue, sales, support, SLA, backlog, route health, shipment, city metrics, operations, devices, incidents
- KPI、趋势、分布、排名、对比、异常、地图热力、资源状态等业务信号

### 2. 页面信息架构
优先提取：
- 指标卡区
- 图表区
- 地图区
- 列表区
- 筛选区
- 时间范围切换区
- drill-down / 明细查看区

重点不是“有图表”，而是要判断：
- 页面如何组织信息
- 哪些区域承担什么职责
- 用户会先看哪里，再看哪里

### 3. 分析动作与交互流
重点看：
- 时间范围切换
- 维度筛选
- 分组对比
- 下钻查看
- hover 详情
- 联动高亮
- 地图区域切换
- tab / segment 切换

如果页面主要是配置录入、表单填写、CRUD 操作，而不是分析洞察，就不应归为 dashboard。

### 4. 指标与图表语义
重点看：
- metric / KPI 名称
- chart 标题
- funnel / line / bar / pie / map / heatmap / table / timeline / distribution
- 环比 / 同比 / 排名 / 占比 / 趋势 / 告警 / SLA 达成率 / backlog aging 等

这些词决定 query 是否专业，不能被抽象成“做一个数据大屏”。

### 5. 面向角色与使用场景
重点判断：
- 这是给谁看的？
- 运营、管理者、分析师、调度员、客服主管、仓配负责人、城市治理人员？
- 用户拿到这个页面后会做什么动作？

### 6. 结果形态与反馈机制
关注：
- 图表更新
- 状态高亮
- 异常提示
- 实时刷新
- drill-down 结果
- 详情卡片
- 地图联动反馈

---

## 类别判断规则
仅当样本核心价值是**信息展示、指标监控、趋势分析、地图可视化、运营洞察、数据决策支持**时，才应归入 dashboard / 可视化类。

典型信号包括：
- KPI、metric、analytics、trend、chart、board、cockpit、monitor、screen、heatmap、map
- 指标卡、图表面板、筛选器、时间切换、联动分析
- 面向运营、管理、分析、调度、监控、决策场景

如果样本主要是：
- CRUD 数据管理
- 表单录入
- 配置台
- 后台管理页面
- 单一图表 demo
- 静态展示落地页
则不应强行判成 dashboard，而要输出 `category_is_dashboard=false`，并补充更合适的 `category1/category2`。

### 类别填写规则
这个 prompt 面向“dashboard 候选样本”，因此：
- 该样本通常不是小游戏，默认应输出 `category_is_dashboard=false`
- 必须填写 `category1`
- 必须填写 `category2`
- 对于真正的 dashboard / 可视化样本，推荐填写：
  - `category1="互动娱乐"`
  - `category2="可视化"`
- 如果 evidence 显示它更像中后台管理、营销页、内容平台等，则应改填对应类别

`category_decision_reason` 必须说明：
- 为什么它不是小游戏
- 为什么它属于可视化 / 或更适合其他非游戏类别

---

## 完整性判断规则
你需要判断该证据是否足以支撑“专业产品需求级别”的逆向。

### 可判为 `is_complete=true` 的情况
满足以下多数条件：
- 能识别明确业务对象
- 能识别核心指标或分析目标
- 能识别页面主结构（指标卡 / 图表 / 筛选 / 地图 / 列表等）
- 能识别至少一种分析动作或交互（筛选、切换、联动、下钻等）
- 能大致判断使用角色或业务场景

### 可判为 `is_complete=false` 的情况
例如：
- 只有单一图表 demo
- 只有空白大屏壳子或占位统计卡
- 只有图表库示例，没有业务语义
- 只有局部模块，无法判断整体分析场景
- 缺少核心指标和业务上下文

**但注意：即使 is_complete=false，也必须生成 query。**

---

## 证据不足时的生成策略
当证据不足时，你仍然必须生成逆向 query，并遵守以下原则：
- 不停止生成
- 不只输出“dashboard / analytics”这类标签词
- 不凭空补复杂组织结构、审批链、后端数据源
- 使用保守但具体的表达
- 优先保留最稳定的信息：业务对象、页面结构、指标词、图表类型、可确认的交互动作

例如可以写：
- “设计一个围绕物流线路状态和延误预警展开的运营可视化看板……”
- “帮我做一个面向客服主管的运营分析面板，用来查看 SLA 达成率、积压趋势和团队表现……”

而不要写：
- “analytics dashboard”
- “metric screen”
- “big screen board”

---

## product_query 写法要求
`product_query` 必须是**一句话产品概念**，要求：
- 用自然中文表达，像真实用户提需求
- 必须写清：服务对象/业务场景 + 监控或分析什么 + 单一核心价值
- 不能只写“做一个 dashboard / 数据大屏”
- 必须体现业务对象

好例子：
- 帮我做一个面向销售管理层的营收分析看板，用来按区域和时间维度查看收入趋势、漏斗转化和重点市场表现。
- 设计一个供应链运营可视化大屏，集中监控运输线路健康度、延误预警和各区域发运状态。
- 我想开发一个客服运营分析面板，帮助主管快速查看 SLA 达成率、积压工单趋势和团队处理效率。

坏例子：
- 做一个数据大屏
- analytics dashboard
- KPI board

---

## feature_query 写法要求
`feature_query` 必须使用 `1. 2. 3.` 数字编号，且尽量覆盖以下内容：

1. **页面结构与信息架构**
   - 指标卡、图表区、地图区、列表区、筛选区、时间切换区

2. **核心指标与图表表达**
   - 需要展示哪些指标
   - 用哪些图表或地图方式呈现

3. **筛选、联动与分析动作**
   - 时间范围、维度筛选、对比分析、下钻、联动高亮、hover 详情等

4. **异常与反馈机制**
   - 告警、状态高亮、趋势变化、同比环比、排名、分布、异常提示

5. **使用角色与分析场景**
   - 面向谁使用
   - 使用者通过这个页面做什么判断或决策

要求：
- 要写具体需求，不要写泛泛总结
- 要像给 AI 开发助手的执行指令
- 不要提“根据代码/根据上传内容/逆向/还原”

---

## queries 写法要求
`queries` 必须输出 2~4 条高质量候选，每条都必须：
- 是完整句子
- 是真实用户会说的话
- 有明确业务对象和分析目标
- 有页面结构或关键分析动作
- 有足够的信息密度

可以使用多样化表达：
- 帮我做一个……
- 我想开发一个……
- 设计一个……
- 创建一个……

不允许输出：
- `analytics dashboard`
- `metric board`
- `data screen`
- 任何“类别名 + 两个特征词”的短句

---

## 输出字段要求
输出必须是**单个 JSON 对象**，不能有 markdown，不能有解释，不能有 code fence。

对象中必须包含以下字段：
- `product_query`
- `feature_query`
- `is_complete`
- `reason`
- `category1`
- `category2`
- `category_is_dashboard`
- `category_decision_reason`
- `queries`

此类样本通常规则如下：
- `category_is_dashboard=false`
- `category1` 与 `category2` 必须填写

---

## category1 / category2 约束
类别必须从以下集合中选择最合适项：

`category1_allowed = ["工程级应用", "内容站点", "互动娱乐"]`

`category2_allowed = ["AI前端应用", "中后台管理", "电商", "社区", "内容平台", "富文本编辑器应用", "图像编辑应用", "视频剪辑工具", "即时通讯", "低代码/流程引擎应用", "跨端应用", "桌面应用", "物联网和嵌入式", "博客/新闻", "营销/着陆页", "作品集/官网", "文档/维基", "单页工具", "小游戏", "互动媒体/艺术", "可视化", "SVG"]`

一级类与二级类映射必须严格遵守：
- `工程级应用` → `["AI前端应用", "中后台管理", "电商", "社区", "内容平台", "富文本编辑器应用", "图像编辑应用", "视频剪辑工具", "即时通讯", "低代码/流程引擎应用", "跨端应用", "桌面应用", "物联网和嵌入式"]`
- `内容站点` → `["博客/新闻", "营销/着陆页", "作品集/官网", "文档/维基"]`
- `互动娱乐` → `["单页工具", "小游戏", "互动媒体/艺术", "可视化", "SVG"]`

推荐：
- 真正 dashboard / GIS / 运营看板类 → `category1="互动娱乐"`, `category2="可视化"`

---

## 输出约束
1. 严禁提及“根据源码 / 根据代码 / 参考附件 / 上传文件 / 逆向 / 还原 / 上面内容 / 输入内容”等来源信息。
2. 严禁输出 markdown。
3. 严禁输出 JSON 之外的解释。
4. 该类样本一般应输出 `category_is_game=false`。
5. 如果 evidence 不足，也必须生成 query，只是表达要更保守。
6. 输出语言使用中文。
7. `feature_query` 必须使用数字编号。
8. `reason` 要简洁说明完整性判断依据。
9. `category_decision_reason` 要基于可观察证据，不要空话。

---

## 一句话提醒
你不是在总结代码，而是在把“可视化 / dashboard 相关 code 证据”转写成一个**值得交给 AI 开发助手执行的数据看板 / 可视化产品需求**。
