# 逆向 Query 的 Code 信息框架（v1）

## 目标
本框架用于指导 LLM 在查看前端 code / 页面证据时，**优先关注哪些信息**，从而稳定生成：
- 信息密度高
- 清晰明确
- 专业度高
- 像真实产品需求的逆向 query

注意：
- **即使证据不足，也仍然要生成逆向 query**。
- 完整性/证据强弱判断的作用，不是决定“是否生成”，而是决定：
  - 生成时的断言强度
  - 细节展开程度
  - 是否使用保守表达
  - 是否需要显式保留不确定性

---

## 一、核心原则

### 1. 看 code 的目的
不是为了做技术分析报告，而是为了回答：
> 如果把这份 code 还原成“用户会提给 AI 开发助手的产品需求”，LLM 最应该抓住什么？

### 2. 逆向 query 的目标产物
逆向出来的 query 应该像：
- 真实用户会提交的产品需求
- 有明确产品对象
- 有清晰主任务
- 有可执行的页面/功能结构
- 有业务语义，而不是空泛类别词

### 3. 证据不足时的处理原则
即使 code 证据不足，也仍要生成 query，但要遵守：
- 不停止生成
- 不因为证据弱就只输出类别标签
- 不无依据扩写复杂后端/业务流程
- 可以保守，但不能空泛
- 应优先使用已确认的页面结构、可见模块、交互动作、业务词汇来构造 query

也就是说：
- **完整性判断影响“怎么生成”**
- **不影响“生不生成”**

---

## 二、LLM 应重点关注的 code 信息层级

按优先级从高到低：

### A. 产品主任务（最高优先级）
首先判断：
- 用户来这个页面/应用，到底是为了完成什么事？
- 这个产品最核心的单一价值是什么？
- 输入是什么？输出是什么？
- 主闭环是什么？

应重点关注：
- 表单提交链路
- 生成/计算/筛选/编辑/导出主流程
- 按钮文案与主 CTA
- 页面中最中心、最大、最频繁调用的功能

输出为逆向 query 时，应体现：
- 产品是什么
- 用户要完成什么
- 系统如何帮助用户完成

---

### B. 页面信息架构
判断页面是如何组织的：
- 顶部导航 / 侧边栏 / 标签页 / 分步结构
- 输入区 / 配置区 / 结果区 / 预览区 / 图表区 / 明细区
- 章节 / 幻灯片 / 分屏 / 卡片网格 / 大屏布局

应重点关注：
- 顶层布局组件
- 页面区域分块
- 路由结构
- tabs / sections / steps / panels

输出为逆向 query 时，应体现：
- 页面主要区域结构
- 各区域承担什么职责
- 是否存在双栏/三栏/分步/大屏等典型信息架构

---

### C. 核心交互流
判断用户如何一步步完成任务：
- 第一动作是什么
- 中间如何修改、切换、追问、筛选、拖拽、编辑
- 最后如何确认、导出、保存、分享、重试

应重点关注：
- 主要事件绑定
- 按钮行为
- 表单状态变化
- 弹窗、侧滑、切换、拖拽、重试、历史记录
- loading / success / error / empty 等状态流转

输出为逆向 query 时，应体现：
- 操作链条
- 状态反馈
- 用户闭环

---

### D. 业务语义与领域词
这是决定 query 是否“专业”的关键层。

应重点抓取：
- 页面中的业务名词
- 模块标题
- 字段名
- 图表标题
- 按钮文案
- 卡片标题
- 标签、筛选器、统计指标名称
- 示例数据中的业务对象

例如：
- support / SLA / backlog / agent workload
- invoice / line item / PDF export
- summary / rewrite / tone preset / follow-up
- route health / shipment status / delay alert
- presenter notes / milestones / walkthrough

输出为逆向 query 时，应优先保留这些业务语义，避免被抽象成：
- dashboard
- tool
- AI app
- presentation
这类空词。

---

### E. 结果形态
判断系统最终产出什么：
- 文本生成结果
- 图表看板
- 演示页面
- PDF / 文件导出
- 游戏结算
- 预览结果
- 卡片化分析结果

应重点关注：
- 输出面板
- 预览区域
- 下载/复制/导出按钮
- chart / map / result card / slide / summary block

输出为逆向 query 时，应体现：
- 用户最终得到什么
- 如何消费这个结果
- 是否支持继续编辑、导出、复制、分享

---

### F. 视觉与交互风格线索
这不是第一优先级，但能帮助 query 更完整。

可关注：
- 大屏/监控风格
- 编辑器式布局
- 对话式布局
- 卡片式工作台
- 沉浸式游戏界面
- slide/deck 叙事感
- 地图/图表主导的大屏视觉

注意：
- 视觉风格只能增强需求，不应取代产品主任务
- 不要把纯样式细节写成需求主体

---

### G. 实现细节中的“产品信号”
不是所有技术实现细节都没用。
以下技术痕迹可能能反推产品能力：
- 路由命名
- 组件命名
- store/state 字段
- action 名称
- API 路径语义
- feature flag 名称
- 上传/下载/导出函数名

例如：
- `generateSummary`, `rewriteDraft`, `exportPdf`, `retryRun`, `applyTonePreset`
- `metricsOverview`, `shipmentMap`, `speakerNotesPanel`

这些名称往往比样式代码更能说明产品意图。

---

## 三、不应过度关注的信息
以下信息通常不应作为逆向 query 的核心：
- 纯样式细节（颜色、阴影、像素级间距）
- 通用工程配置
- 打包工具、lint、测试框架
- 无业务语义的底层 hooks
- 纯复用组件内部实现
- 与主任务弱相关的动画技术实现

例外：
如果某些样式/动画本身承载了产品核心体验（如展示型叙事、游戏反馈、大屏监控态），则可作为补充信号。

---

## 四、证据不足时也要生成 query 的策略

### 1. 可生成，但要保守
当证据不足时，query 仍应保留：
- 可识别的产品对象
- 最可能的主任务
- 可确认的页面结构
- 可确认的交互动作

### 2. 降低断言强度
可以使用：
- “支持……”
- “围绕……展开”
- “提供……能力”
- “主要包含……”

避免使用：
- 过于具体但证据不足的复杂业务流程
- 虚构角色体系、审批链、数据库逻辑
- 凭空补后端能力

### 3. 缺证据时优先保留的东西
优先级如下：
1. 页面主任务
2. 主要区域结构
3. 明显交互动作
4. 业务语义词
5. 输出形态

## 五、推荐的结构化字段清单（evidence schema v1）

建议从 code / 页面证据中抽取如下字段，供后续 LLM 逆向 query 使用：

### 基础标识
- `sample_id`
- `source_path`
- `title`
- `url`（如有）
- `preset_category`（如有）

### 产品意图层
- `product_goal`：产品要解决什么问题
- `target_user_or_role`：面向谁
- `core_value`：最核心价值是什么
- `primary_workflow`：主任务闭环
- `input_object`：用户输入什么
- `output_object`：系统输出什么

### 页面结构层
- `main_layout_regions`：主要区域结构
- `navigation_pattern`：导航/路由/steps/tabs/sections
- `core_modules`：核心模块列表
- `result_area_type`：结果区域类型（文本/图表/地图/预览/卡片等）

### 交互行为层
- `key_interactions`：关键动作
- `state_transitions`：空态、加载、成功、失败、重试等
- `editable_controls`：可编辑项 / 参数项
- `final_actions`：导出、复制、保存、分享、提交等

### 业务语义层
- `domain_terms`：高价值业务词
- `metric_terms`：指标/图表/分析词
- `cta_terms`：关键按钮/动作文案
- `entity_terms`：对象名词（订单、发票、客服回复、演示章节等）

### 证据质量层
- `completeness_level`：high / medium / low
- `completeness_reason`
- `integrity_flags`
- `confidence_notes`

### 类别辅助层
- `candidate_category`
- `reclassified_category`
- `category_reason`

## 六、LLM 逆向 query 生成时的使用建议

### 高完整性样本
应生成：
- 明确的一句话产品需求
- 结构化功能需求
- 2~4 条高信息密度 query 候选

### 中完整性样本
应生成：
- 保守但具体的产品需求
- 重点抓主任务 + 主界面 + 可确认交互
- 避免过度补细节

### 低完整性样本
也必须生成，但应：
- 聚焦最稳定的产品对象和任务
- 不写复杂流程
- 允许 query 更简洁，但仍然是“真实需求句”，而不是标签词

---

## 七、给后续 prompt 设计的直接结论

后续 prompt 不应只问：
- “这是什么类别？”
- “请生成 query”

而应先引导模型抽取：
1. 主任务
2. 页面结构
3. 核心交互
4. 业务语义
5. 输出结果
6. 证据强弱

然后再据此生成 query。

也就是说，推荐采用两层设计：
- **层 1：code / evidence 关注点抽取**
- **层 2：基于关注点生成逆向 query**

这样更容易稳定得到高质量产品需求，而不是模板化类别描述。

---

## 八、一句话结论

逆向 query 的关键，不是把 code “总结一下”，而是从 code 中稳定抽出：
**主任务、页面结构、交互闭环、业务语义、输出结果、证据强弱**，再据此生成始终可用、但断言强度可调的产品需求 query。
